### B4 — "What to do" section     ❌

User's headline complaint, only half-fixed. Verbatim:

```
What to do


Acceptance (skillslab check)
```

That's the entire section. **Header renders, body is empty.**

Root cause (verified in the cached file): `step_to_markdown` injects the `## What to do` heading then dumps the source content as one single-line HTML blob — `<h3>...</h3><pre><code>...</code></pre><p>Expected:...</p><details><summary>...</summary>...</details>` — all on ONE line with no newlines. Rich's `Markdown` widget treats inline HTML as escape-text and silently drops it. Beginner sees "here's what to do" followed by nothing — **worse than no section at all.**

Fix: in `step_to_markdown`, run the already-imported `html_to_markdown` on `instructions` before emitting (or walk with BeautifulSoup and emit numbered markdown bullets with fenced code blocks).

Clean, beginner-readable, exactly what the user asked for. Win.

### B6 — Other rendering weirdness (beginner perspective)     ❌

**(a)** The top Rich panel header is partially blank: `module:     type: terminal_exercise    step id: None` — empty `module:` value and `step id: None`. Three lines later the body header has the correct values (`step id: 85163`). Looks like a null-handling bug. Root cause: panel reads from `_load_cursor`'s stripped step dict; body reads from the markdown file which has the fields.

**(c)** Visual hierarchy is noisy — a Rich Panel, body header, 80-char `---` separator, then another section before any actionable content. One separator could be dropped.

## 5. Section D — Browser URL correctness

### D1 — Web step browser URL     ❌  ❌

Two separate broken paths.

**(a) `skillslab dashboard --course kimi`**:

```
  http://host.docker.internal:8001/#created-698e6399e3ca
```

**Still prints `host.docker.internal`**. This is the exact bug from user-filed issue #1. The fix landed for `_print_web_pointer` (which uses `state.web_url()`) but `dashboard` at `cli.py:1022` calls `state.api_url()` directly:

```python
url = f"{state.api_url()}/#{cid}"
```

Should be `state.web_url()`. **One-line fix.** I'd be very confused as a beginner — I'd copy-paste this into Chrome and get DNS-doesn't-resolve.

**(b) `skillslab spec` web pointer (M0.S1, web step)**:

```
│ Open in your browser:
│   /#created-698e6399e3ca
```

**Host is missing entirely.** Root cause: the supplied `cli/docker-compose.yml` always passes `SKILLSLAB_WEB_URL=${SKILLSLAB_WEB_URL:-}` — an *empty string* — into the container. `state.web_url()` checks `if "SKILLSLAB_WEB_URL" in os.environ:` which is True for empty strings, then returns `os.environ["SKILLSLAB_WEB_URL"].rstrip("/")` = `""`. The auto-derive-from-`host.docker.internal` fallback never runs.

**Fix:** in `state.web_url()`, change the guard to `if os.environ.get("SKILLSLAB_WEB_URL"):` (truthy check) so empty strings fall through to the auto-derive branch. **One-line fix.**

### D2 — `SKILLSLAB_WEB_URL` env override     ⚠ (partial)

Setting `SKILLSLAB_WEB_URL=https://lms.example.com` works for `spec`'s web pointer:

```
│ Open in your browser:
│   https://lms.example.com/#created-698e6399e3ca
```

But NOT for `dashboard` (because `dashboard` ignores `web_url()` — see D1a above):

```
$ SKILLSLAB_WEB_URL=https://lms.example.com skillslab dashboard --course kimi
  http://host.docker.internal:8001/#created-698e6399e3ca
```

Same root cause. Fix `dashboard` to call `state.web_url()` and the override will work end-to-end.

- **`skillslab now`** — single command, no flags, answers "where am I and what do I do?". The kind of thing kubectl/aws lack.
- **The launcher script** — symlink-resolving `realpath`, sensible `SKILLSLAB_REPO` override, well-commented. Installs in 30 seconds.

- **`skillslab next` CTA panel** — bordered, bold-cyan command, explicit "then:" sub-line.

## 9. Suggestions for the permanent CLI-walk harness

Things the harness should ALWAYS check on every PR:

1. **Surface-aware URL audit.** Grep ALL command output for `host.docker.internal` — should never appear. (Catches D1a + D1b.)
2. **Empty-string env-var trap.** Run with `SKILLSLAB_WEB_URL=` explicitly empty; URLs must still be well-formed. The compose file makes this the *default* case.
3. **Cached vs fresh render parity.** Diff `spec` output BEFORE and AFTER `skillslab start <slug>` — divergence = stale cache.
4. **Section-emptiness check.** Grep `spec` output for `## Heading` immediately followed by another `## Heading`. Empty-section bug.
5. **Help text doesn't leak maintainer-speak.** Lint for `User-filed`, `2026-`, `(was buggy, fixed)`, `Per Opus`. None belong in user help.
6. **Panel/dim parity for next-action hints.** All of `status`/`next`/`spec`/`now`/post-`check` should use the same Rich Panel — assert no `[dim]skillslab \w+[/dim]` flat text.
7. **`/work` cleanliness** — with launcher: empty. Without launcher (compose from `cli/`): polluted. Both confirm the launcher is still needed AND works.
8. **`courses` slug coverage.** Fail if too many listed courses show `—` for slug.
9. **First-run UX simulation.** Wipe token, run `skillslab status` — must point to `login` without a Python traceback.
10. **Help-text wrapping check.** Regex `skillslab \w+\s{2,}skillslab \w+` against `--help` to catch lines mashed together (the `now --help` glitch).
